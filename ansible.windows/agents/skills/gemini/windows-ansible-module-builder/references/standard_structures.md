# Ansible Collection Standards (Windows/Hyper-V)

Based on established patterns from `ansible.windows` and `microsoft.hyperv`, this guide defines the architectural standards for developing high-quality Ansible collections for Windows.

## 2. The Split-Plugin Pattern (Windows)

Windows modules MUST be split into two files to support Ansible's execution model:

### A. The Metadata File (`.py`)
- **Location:** `plugins/modules/module_name.py`
- **Content:** Standard Python file containing ONLY the `DOCUMENTATION`, `EXAMPLES`, and `RETURN` strings. 
- **Purpose:** Enables `ansible-doc` to read metadata and provides a hook for Ansible's module loader.

### B. The Logic File (`.ps1`)
- **Location:** `plugins/modules/module_name.ps1`
- **Content:** Pure PowerShell logic.
- **Requirement:** Must use `[Ansible.Basic.AnsibleModule]::Create($args)` to handle input/output.

---

## 3. Module Utilities (`module_utils`)

To prevent code duplication, shared logic must be extracted into `plugins/module_utils/`. 
- **Naming:** Prefer PascalCase for PowerShell modules (e.g., `HyperV.psm1`).
- **Standard Functions:** Utility modules should provide consistent ways to:
  - Map Ansible parameters to PowerShell Cmdlet parameters.
  - Test if properties have changed (Idempotency).
  - Format output objects for Ansible.

---

## 4. Integration Testing Strategy

Every module requires a corresponding integration target in `tests/integration/targets/`.

### The 4-Stage Test Loop:
1. **Initial Change:** Verify the module performs the action and returns `changed: true`.
2. **Idempotency:** Run the exact same task again. Verify it returns `changed: false`.
3. **Check Mode:** Run with `check_mode: true`. Verify it reports what *would* change without actually doing it.
4. **Error Handling:** Intentionally provide bad input or trigger a failure state. Verify the module fails gracefully with a helpful message.

### Metadata Dependencies:
Use `meta/main.yml` in the test target to pull in common setup roles (like `setup_hyperv` or `ansible.windows.setup_winrm`) to ensure the environment is ready for the module being tested.
